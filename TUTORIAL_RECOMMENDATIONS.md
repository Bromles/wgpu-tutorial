# Рекомендации по дальнейшей структуре руководства

Этот файл фиксирует идеи по развитию руководства.

## Принципы подачи

Руководство ориентировано на опытных Rust-разработчиков, не знакомых с графикой. Основные принципы:

1. **Один концепт — один визуальный результат.** Каждая глава добавляет одну новую идею, и читатель сразу видит
   результат на экране. Не должно быть глав, заканчивающихся тем же, чем они начинались.

2. **Теория — ровно в тот момент, когда она нужна.** Не заранее, не потом. Математика появляется перед первой
   3D-трансформацией, bind groups — перед первой передачей данных в шейдер.

3. **Минимум абстракций до первого результата.** Три главы до треугольника — это много. LearnOpenGL выводит зелёный
   экран за несколько абзацев. Чем быстрее читатель увидит результат, тем лучше.

4. **WGSL — полноценный язык, а не справочник.** Rust-разработчику WGSL кажется знакомым, но в нём достаточно ловушек
   (layout, alignment, @binding, отличия в типах). Нужна систематическая глава с визуальным результатом.

5. **Каждая глава отвечает на вопрос «что я увижу в конце?».** Если ответа нет — главу стоит пересмотреть.

## Структура маршрута

### 1. Начало работы

```
Создание окна
  └ visual result: пустое окно

Инициализация wgpu
  └ visual result: зелёный экран
  └ теория: Instance, Adapter, Device, Queue, Surface, RenderPass
  └ принцип: сократить теорию, быстрее к результату

Первый треугольник
  └ visual result: тёмно-красный треугольник на зелёном фоне
  └ вводится: framework, WGSL (минимум), RenderPipeline
```

Первые три главы показывают полный код вручную. Framework вводится в главе про треугольник как секция, а не отдельная
глава.

### 2. Модель данных GPU

Это **ключевое отличие от LearnOpenGL**: в wgpu без bind groups нельзя передать в шейдер вообще ничего — ни матрицу
камеры, ни текстуру. Поэтому модель данных GPU должна идти до 3D.

```
Шейдеры и WGSL
  └ visual result: цветной треугольник (вершины разных цветов)
  └ систематический разбор: типы, функции, @binding, control flow, отличия от Rust
  └ vertex buffer с цветами

Вершинные буферы
  └ visual result: прямоугольник из двух треугольников
  └ визуальный скачок от треугольника к прямоугольнику

Индексные буферы
  └ visual result: тот же прямоугольник, но эффективнее
  └ оптимизация: зачем, как, когда имеет смысл

Uniform и bind groups
  └ visual result: цвет, меняющийся во времени (анимация)
  └ концепция: uniform buffer → bind group layout → bind group → @group(0) @binding(0)
  └ это фундамент для всего последующего: камера, текстуры, lighting
  └ здесь вводится encase: для uniform-буферов WGSL требует 16-байтовое выравнивание структур,
    вручную считать паддинг мучительно. encase через ShaderType решает это автоматически.
    bytemuck остаётся для вершинных данных (VertexBufferLayout позволяет задать смещения вручную).
    glam уже в workspace с фичей encase — mat4, vec3 реализуют ShaderType из коробки.

Текстуры и сэмплеры
  └ visual result: текстурированный прямоугольник
  └ создание текстуры, загрузка изображения, sampler, UV-координаты

Storage buffers
  └ visual result: например, compute-подсчёт или передача массива данных
  └ если не требуется сразу — можно отложить до compute-секции
```

### 3. Математика для графики

**Ставится ПЕРЕД 3D**, не после. Без векторов и матриц трансформации и камера — магия.

```
Векторы, матрицы, операции
  └ не справочник, а ровно то, что нужно для следующей главы
  └ можно浓缩 до одной страницы с примерами

Система координат WebGPU
  └ clip space, NDC, экранное пространство
  └ откуда берутся координаты от -1 до 1
```

### 4. 3D и камера

```
Трансформации MVP
  └ visual result: вращающийся куб
  └ model → view → projection, зачем каждая

Depth buffer
  └ visual result: несколько кубов без взаимопроникновения
  └ раньше было невозможно, теперь — корректная глубина

Камера
  └ visual result: свободный полёт (WASD + мышь)
  └ нужна интеграция input в framework (update trait method)

Instancing
  └ visual result: сотня кубов одним draw call
  └ визуально впечатляющий скачок
```

### 5. Освещение

```
Нормали и базовый свет
  └ visual result: освещённый куб, reacting к позиции источника

Материалы и множественные источники
  └ visual result: несколько источников света разных цветов

Gamma, sRGB, linear color
  └ conceptual page, linked from here

Normal maps
  └ visual result: детализация поверхности без лишней геометрии

Тени
  └ visual result: объекты отбрасывают тени
```

### 6. Продвинутый рендер

```
Render-to-texture, постпроцессинг
  └ visual result: blur / grayscale / инверсия поверх сцены

HDR, tone mapping
  └ visual result: яркие источники света не выжигают сцену

MSAA
  └ visual result: гладкие края без лесенки

Deferred rendering
  └ visual result: сотни источников света без падения производительности

Compute passes
  └ visual result: compute-фильтр, particles, или GPU culling
```

### 7. PBR Track

```
Metallic-roughness
  └ visual result: реалистичные материалы

IBL, environment maps
  └ visual result: отражения окружения

glTF loading
  └ visual result: загруженная модель с материалами

Mini-viewer
  └ финальный проект: просмотрщик glTF-моделей
```

## Учебный каркас

### Что уже сделано

Framework реализован в `code/framework/`:

```rust
pub trait Example: 'static {
    fn init(ctx: &GpuContext) -> Self;
    fn resize(&mut self, ctx: &GpuContext, new_size: PhysicalSize<u32>);
    fn render(&mut self, ctx: &GpuContext, view: &TextureView, encoder: &mut CommandEncoder);
}
```

```rust
pub struct GpuContext {
    pub device: Device,
    pub queue: Queue,
    pub surface: Surface<'static>,
    pub surface_config: SurfaceConfiguration,
    pub surface_format: TextureFormat,
}
```

- `pollster::block_on` для async-вызовов
- hello-triangle переписан на framework (~90 строк вместо ~340)

### Как не спрятать wgpu за каркасом

Каркас должен экономить внимание, а не становиться источником знания. Главный принцип: читатель должен понимать,
какие именно строки wgpu-кода были спрятаны и как их вернуть в своё приложение.

Правила:

- `GpuContext` использует публичные поля `wgpu`-типов, никаких обёрток
- В hello-triangle есть секция «Что именно скрывает каркас» с полным списком
- Для важных тем (depth buffer, camera, textures, render-to-texture, compute) — блок «без framework» с переносным
  списком действий
- Периодически — standalone-примеры без каркаса (`hello-triangle-standalone`, `texture-standalone`)

### Каркас должен быть намеренно скучным

Не добавлять в framework:

- ECS, asset manager, scene graph, material system, renderer graph, DI, generic abstraction layers

Допустимый минимум:

- окно, surface lifecycle, device/queue, resize, frame acquire/present, timing

Input (клавиатура/мышь) — добавить когда дойдёт до главы про камеру.

## Формат кода для глав

Для каждой темы — три артефакта:

```text
article.md              // объяснение
code/.../final          // состояние после главы
code/framework          // общий учебный runtime
```

В статье — inline-код (не include-фрагменты, они ненадёжны). Полный код — по ссылке на репозиторий.

## Главный вывод

Курс должен быть похож на LearnOpenGL: **последовательный и практичный**, каждая глава даёт визуальный результат,
теория появляется ровно когда нужна. Но при этом учитывает специфику wgpu: bind groups, layout/alignment и WGSL —
необходимый фундамент, без которого невозможно двигаться дальше.
